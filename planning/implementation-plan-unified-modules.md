# SAFE Inc. Implementation Plan — Documents & Redaction + Project Management + Accounting Coherence

**Status**: Phase Integration Sequencing  
**Last Updated**: 2026-04-20  
**Priority**: Module integration → Accounting coherence fixes

---

## Executive Summary

This plan integrates three interconnected workstreams into SAFE Inc:

1. **Phase 1 (Week 1-2)**: Database migration to unified schema
2. **Phase 2 (Week 2-4)**: Documents & Rédaction module (14 tasks, Tiptap + AI)
3. **Phase 3 (Week 4-6)**: Gestion de Projet & Calendrier module (16 tasks, automation + BullMQ)
4. **Phase 4 (Week 6+)**: Accounting coherence fixes (multi-source synchronization, transactional integrity)

**Key architectural decisions**:
- Dossier-centric scope for all features (except Calendar, which is firm-wide)
- Mandatory "force-active-task" rule prevents dossier closure without active task
- Transactional synchronization for accounting data sources (Invoices, TimeEntries, Debours, TrustTransactions)
- Tiptap JSON + Versioning for RedactionDocument (separate from Document model for uploads)
- BullMQ + Redis for automation rule evaluation and background job scheduling

### Task 3.7: Automation rule evaluation engine

**Objective**: Implement periodic rule checking for inactivity, force-active-task, deadline reminders

**Dependencies**: Task 3.1, 3.2 complete

**Acceptance Criteria**:
- Engine at `lib/automation/evaluateRules.ts`
- Rule types:
  1. **INACTIVITY_ALERT**: if Dossier.lastActivityAt + threshold < now, trigger CASE_INACTIVE notification
  2. **FORCE_ACTIVE_TASK**: if Dossier.tasks.none(status=IN_PROGRESS), create auto-generated task "Review case status"
  3. **PRE_DEADLINE_REMINDER**: 1 day before task dueDate, send TASK_DUE_SOON notification
  4. **AUTO_TASK_AFTER_COMPLETION**: when Task.status=COMPLETED, auto-create next task (chained task)
  5. **PRACTICE_SPECIFIC**: custom rules by practice area (e.g., "Family law: check for support calculations every 30 days")

**Rule execution flow**:
```
1. CronJob triggers at 08:00 daily (configurable)
2. FOR each Cabinet with isActive AutomationRules:
   3. FOR each Rule.scope (FIRM, USER, CASE_TYPE, CLIENT):
      4. Query matching Dossiers
      5. FOR each Dossier:
         6. Evaluate rule.config (e.g., "if lastActivityAt + 14 days < now")
         7. IF rule triggered:
            8. Create RuleTrigger record
            9. Execute action (create Task, send Notification)
            10. Queue notification jobs
3. Log results
```

### Task 3.8: "Force active task" rule enforcement

**Objective**: Prevent dossier closure without active task (mandatory rule)

**Dependencies**: Task 3.7 complete

**Acceptance Criteria**:
- On each Dossier.statut update (to cloture):
  - API checks: are there any Task.status = IN_PROGRESS?
  - If NO → reject update, return 400 with message "Dossier cannot be closed without an active task"
  - If YES → allow closure
- UI: "Close dossier" button disabled if no active task
- Tooltip explains rule
- Admin override: avocat responsable can force-close (audit logged)

**Implementation**:
```typescript
// api/dossiers/:id [PUT]
async function updateDossierStatus(req, res) {
  const { statut } = req.body;

  if (statut === 'cloture') {
    const activeTasks = await prisma.task.findMany({
      where: { dossierId: req.params.id, status: 'IN_PROGRESS' }
    });

    if (activeTasks.length === 0) {
      return res.status(400).json({
        error: 'Dossier cannot be closed without an active task',
        code: 'FORCE_ACTIVE_TASK_REQUIRED'
      });
    }
  }

  // Proceed with update
}
```

### Task 4.2: Implement billing synchronization service

**Objective**: Create transactional accounting updates

**Dependencies**: Task 4.1 complete

**Acceptance Criteria**:
- Service at `lib/accounting/syncBilling.ts`
- Function: `recalculateInvoice(invoiceId)`
  - Fetch all TimeEntry items linked to invoice
  - Recalculate montantFinalDue based on dossier modeFacturation
  - If modeFacturation = horaire: sum(TimeEntry.duration * TimeEntry.hourlyRate)
  - If modeFacturation = forfait: fetch ForfaitService.montant
  - If modeFacturation = retainer: deduct from Client.retainerBalance
  - Update Invoice.montantFinalDue
  - Trigger Payment reconciliation
  - Atomic transaction: all-or-nothing

**Example**:
```typescript
async function recalculateInvoice(invoiceId: string) {
  return prisma.$transaction(async (tx) => {
    const invoice = await tx.invoice.findUnique({ where: { id: invoiceId } });
    const dossier = await tx.dossier.findUnique({ where: { id: invoice.dossierId } });

    let newAmount = 0;
    if (dossier.modeFacturation === 'horaire') {
      const entries = await tx.timeEntry.findMany({
        where: { invoiceId },
        include: { user: true }
      });
      newAmount = entries.reduce((sum, e) => sum + (e.duration * (e.hourlyRate || e.user.defaultHourlyRate)), 0);
    }
    // ... other modes

    await tx.invoice.update({ where: { id: invoiceId }, data: { montantFinalDue: newAmount } });
    
    // Cascade: recalculate outstanding balance
    const payments = await tx.payment.findMany({ where: { invoiceId } });
    const newOutstanding = newAmount - payments.reduce((s, p) => s + p.montant, 0);
    
    return { invoice: { ...invoice, montantFinalDue: newAmount }, outstanding: newOutstanding };
  });
}
```

## Risk Mitigation

### Schema Migration Risk
**Risk**: Migration fails on production, data corruption.  
**Mitigation**:
- Extensive staging tests (use production backup)
- Rollback plan: prior migration reversal script
- Monitoring during migration: real-time error logging
- Maintenance window: 02:00-04:00 UTC (low traffic)

### Accounting Data Loss Risk
**Risk**: Sync overwrites correct invoice amount with calculated wrong amount.  
**Mitigation**:
- Full audit trail (before/after snapshots)
- Manual override always available
- Automatic detection: variance > 5% → block sync, escalate
- Reconciliation report generated before any destructive sync

### BullMQ Job Loss Risk
**Risk**: Jobs lost if Redis crashes, automations don't trigger.  
**Mitigation**:
- Redis persistence (RDB snapshots + AOF)
- Dead-letter queue for failed jobs (retries, then manual review)
- Monitoring: alert if queue depth exceeds threshold
- Fallback: daily batch job re-triggers missed automations
